## Plan: 第1-2周 vLLM 执行任务清单
> **状态**：第1-2周任务已全部完成。实际使用 **Qwen2.5-3B-Instruct-AWQ**（AWQ 4-bit 量化）替代原计划的 7B fp16 模型，因 RTX 4060 Laptop 8GB 显存不足以加载 7B fp16。以下内容中提到的 "7B" 均为原始计划描述，实际实验配置以 `configs/baseline_0.json` 为准。
目标是把“vLLM 推理性能优化”的开题要求拆成一套可直接交给写代码智能体执行的顺序任务清单。整体策略是先设计一个最小但完整的实验工程骨架，再按“研究问题定义 → 环境可运行 → 服务可调用 → 压测可统计 → baseline 0 可复现”的顺序推进，确保前两周一定能产出第一份可信基准结果。

**Steps**
1. Phase 0：确定最小工程范围。先把本阶段目标限制为单卡 GPU、单模型、单套 baseline 场景，不进入多模型横评、多卡部署、LoRA、量化对比或自动调参。执行者需要先阅读开题报告并提炼一句话研究问题，再把本阶段的纳入项和排除项写成说明文档。产出应包括“研究目标”、“本阶段回答的问题”、“本阶段不处理的问题”。
2. Phase 1：设计项目目录骨架。让写代码智能体先在工作区建立统一结构，至少预留 docs、configs、scripts、benchmarks、results、logs 六类目录，用于放说明文档、实验配置、环境脚本、压测脚本、结果文件和运行日志。该步骤只做骨架，不追求功能完整。*depends on 1*
3. 为每个目录定义用途与命名规则。docs 用于研究说明和实验记录模板；configs 用于模型、服务和压测参数；scripts 用于环境安装、服务启动和结果收集； benchmarks 用于请求样本和压测驱动； results 用于保存 baseline 输出； logs 用于保存服务端和压测端日志。执行者需要同时写一份简短的目录说明，避免后续文件乱放。*depends on 2*
4. Phase 2：收敛研究问题与参数边界。写代码智能体不需要替你写整篇文献综述，但需要先生成一个实验说明模板，明确固定变量和可变变量。固定变量建议包含 GPU 型号、显存、Python 版本、CUDA 版本、vLLM 版本、模型名称、采样参数；可变变量只保留并发数、输入长度、输出长度、请求速率。*depends on 1*
5. 定义 baseline 0 的最小实验场景。要求执行者在配置文件中固定一套可运行参数，例如固定模型为 7B 指令模型、固定 prompt 长度区间、固定 max tokens、固定 temperature、固定一个并发档位和一次请求总量。这个配置的目标不是最优，而是第一次稳定跑通。*depends on 4*
6. Phase 3：制定硬件与软件信息采集方案。让执行者准备一份系统信息采集脚本或记录模板，至少收集 GPU、CPU、内存、操作系统、驱动版本、CUDA 版本、Python 版本、PyTorch 版本、vLLM 版本。该信息必须自动保存到 results 或 docs 中，作为基准结果的一部分。*parallel with 5*
7. Phase 4：准备环境搭建文档与脚本。写代码智能体需要设计安装流程，建议拆成“环境准备”、“依赖安装”、“模型准备”、“服务启动验证”四段。脚本层面至少需要一个环境初始化脚本和一个服务启动脚本；文档层面需要说明执行顺序、前置条件和失败排查入口。*depends on 2*
8. 设计依赖版本锁定策略。为了保证结果可复现，执行者应在配置或文档中写清 Python 版本、PyTorch 版本、CUDA 兼容范围和 vLLM 安装方式，避免使用“最新版本”这种不可复现实验描述。*depends on 7*
9. Phase 5：设计服务配置文件。要求写代码智能体把模型名、host、port、dtype、tensor parallel size、max model len、gpu memory utilization 等核心启动参数放入配置文件或统一脚本参数中，而不是散落在命令行历史里。这里的目标是“同一份配置可以重复启动同一服务”。*depends on 5, 7*
10. 设计服务可用性验证步骤。执行者需要实现一次最小 API 调用验证，验证项包括：服务是否成功启动、接口是否可访问、返回是否包含期望文本、异常时日志是否可追踪。建议将该验证做成单独脚本，便于后续重复执行。*depends on 9*
11. Phase 6：设计压测方案。写代码智能体需要先选定一条简单且稳定的压测路线，优先满足“能发请求、能记录吞吐与时延、能保存原始结果”三个条件。前两周不要求压测框架复杂，重点是统计口径清晰。*depends on 10*
12. 定义压测输入集与请求模板。要求执行者准备一小组固定请求样本，覆盖 baseline 0 需要的 prompt 场景，并把输入样本单独保存，避免后续因为输入内容变化导致结果不可比。请求模板中应固定模型名、消息格式、最大输出长度和采样参数。*depends on 11*
13. 设计压测结果输出格式。压测完成后至少要输出吞吐量、平均时延、P95 时延、首 token 时延、总请求数、成功率、测试持续时间。结果应同时保存为机器可读格式和人可读摘要，方便后续制表和画图。*depends on 11*
14. Phase 7：设计 baseline 0 执行流水线。要求执行者把流程串成固定顺序：启动服务、等待健康检查通过、执行最小 API 验证、运行 baseline 压测、收集硬件与环境信息、汇总结果、写入结果目录。该流水线可以先是手动触发的多个脚本，不要求一开始就做成全自动。*depends on 6, 10, 13*
15. 设计失败回滚与排查点。为了让写代码智能体更稳妥执行，需要明确每一步失败时看哪里：环境安装失败看依赖日志；服务启动失败看启动日志和显存占用；API 验证失败看接口地址和模型加载状态；压测异常看请求格式和超时设置。这个排查说明要写进 docs。*parallel with 14*
16. Phase 8：定义第1周和第2周的节奏。第1周目标是研究问题收敛、项目骨架完成、环境脚本和服务启动跑通；第2周目标是压测链路跑通、baseline 0 完成、阶段材料沉淀。这样可以避免两周内一直停留在环境阶段而没有实验结果。*depends on 14*
17. 第1周任务拆分。第1-2天完成开题问题提炼与目录搭建；第3-4天完成环境安装文档、版本锁定和模型准备；第5-7天完成 vLLM 服务启动脚本和 API 验证脚本。每个子任务都要有明确输出文件。*depends on 16*
18. 第2周任务拆分。第8-10天完成压测脚本、请求样本和结果导出；第11-12天完成 baseline 0 正式运行和结果校验；第13-14天整理实验记录、结果表和周报/开题支撑材料。*depends on 17*
19. Phase 9：定义交付物。写代码智能体本阶段至少要交付：项目目录骨架、环境说明、环境安装脚本、服务启动脚本、服务验证脚本、压测脚本、baseline 配置文件、硬件信息记录结果、baseline 0 结果文件、阶段说明文档。只要这十类交付物齐全，本阶段就算闭环。*depends on 18*
20. Phase 10：定义后续扩展接口。为了方便第3周以后继续做参数扫描，执行者在写配置和脚本时要预留模型名、并发、输入长度、输出长度和请求速率等可配置入口，但当前只需要先填充 baseline 0 默认值。*parallel with 19*

**具体执行指令（交给写代码智能体）**
1. 先阅读现有开题报告，输出一份简短的研究问题说明文档，明确本阶段目标、约束、核心指标和排除范围。
2. 在工作区创建实验工程目录骨架，并写一份目录用途说明文档。
3. 新建 baseline 配置文件，固定单卡 GPU、7B 模型、最小 prompt 设定和默认压测参数。
4. 新建环境准备文档和安装脚本，锁定关键版本，保证环境可复现。
5. 新建 vLLM 服务启动脚本，把核心服务参数集中管理。
6. 新建服务健康检查或最小调用验证脚本，保证服务可用性能被快速确认。
7. 新建请求样本文件和压测脚本，输出统一格式的性能指标。
8. 新建硬件与软件环境信息采集脚本或记录机制，并把结果落盘。
9. 设计 baseline 0 执行顺序文档，说明先后步骤、输入输出和失败排查点。
10. 完成首轮 baseline 0 后，把结果整理成摘要文档和结果表，为后续周报与开题材料复用。

**建议生成的文件与目录**
- d:/vlllm/docs — 存放研究问题说明、目录说明、环境文档、实验记录模板、阶段总结
- d:/vlllm/configs — 存放 baseline 配置、服务配置、压测配置
- d:/vlllm/scripts — 存放环境安装、服务启动、健康检查、系统信息采集等脚本
- d:/vlllm/benchmarks — 存放请求样本、压测驱动、结果汇总逻辑
- d:/vlllm/results — 存放 baseline 0 原始结果、汇总结果、硬件环境记录
- d:/vlllm/logs — 存放服务日志与压测日志
- d:/vlllm/CS2207-刘沛林开题报告.doc — 作为研究目标和阶段要求的源材料

**Verification**
1. 结构验证：工作区目录清晰，文档、配置、脚本、结果和日志分区明确。
2. 研究问题验证：存在一份简明文档，能明确说明研究问题、参数边界、核心指标和不纳入范围的内容。
3. 环境验证：按照文档执行后，vLLM 服务可启动并可接受一次真实请求。
4. 压测验证：压测脚本能跑完一轮固定请求，并输出结构化性能结果。
5. baseline 0 验证：存在一组固定配置实验的完整记录，包含环境信息、配置、日志、结果表和摘要。
6. 交接验证：把任务单独交给写代码智能体后，对方无需再猜测目录设计、文件职责和执行顺序。

**Decisions**
- 研究方向固定为 vLLM 推理性能优化。
- 实验硬件固定为单卡 GPU 工作站。
- baseline 0 先采用 7B 级模型，优先确保流程可运行和结果可复现。
- 第1-2周的重点是工程骨架和首个基准结果，不是性能优化结论本身。
- 本计划默认当前工作区几乎为空，需要从零搭建实验工程。

**Further Considerations**
1. 如果你的开题报告已经明确指定模型家族，执行时应优先服从开题要求，而不是采用通用 7B 推荐模型。
2. 如果你的单卡显存偏小，baseline 0 配置应优先保守，先保证能稳定加载与响应，再逐步抬高参数。
3. 如果你准备把这份任务单直接贴给写代码智能体，建议先让它按“目录骨架 + 文档 + 配置 + 脚本”顺序执行，不要一开始就直接冲 baseline 压测。

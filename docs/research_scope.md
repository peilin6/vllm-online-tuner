# 第 1-2 周研究问题说明

## 研究方向

**vLLM 在线推理服务性能评测与优化**

## 本阶段要回答的问题

> 在固定硬件（单卡 GPU）和固定模型（3B 级量化指令模型）条件下，vLLM 在线推理服务在不同请求压力下的吞吐量、首 token 时延和端到端时延表现如何？

## 核心评价指标

| 指标 | 缩写 | 单位 | 说明 |
|------|------|------|------|
| 吞吐量 | Throughput | req/s | 单位时间内完成的请求数 |
| 首 token 时延 | TTFT | ms | 从发送请求到收到第一个 token 的时间 |
| 平均端到端时延 | Avg Latency | ms | 请求发出到完整响应接收的平均耗时 |
| P95 端到端时延 | P95 Latency | ms | 95% 请求的端到端耗时上界 |
| GPU 显存占用 | GPU Mem | MiB | 推理过程中的峰值显存使用量 |
| GPU 利用率 | GPU Util | % | 推理过程中 GPU 计算单元使用率 |

## 固定变量（本阶段不变）

| 变量 | 本阶段取值 |
|------|-----------|
| GPU 型号 | 由实际硬件决定，运行时自动采集 |
| 显存容量 | 由实际硬件决定 |
| Python 版本 | 3.10 |
| CUDA 版本 | 12.1+ |
| vLLM 版本 | 0.6.x（以实际安装为准，锁定在 requirements 中）|
| 模型名称 | Qwen/Qwen2.5-3B-Instruct-AWQ |
| 量化方式 | AWQ 4-bit |
| 采样参数 | temperature=0.7, top_p=0.9, max_tokens=256 |
| Tensor Parallel | 1（单卡） |

## 可变变量（后续扫描，本阶段只取默认值）

| 变量 | baseline 0 默认值 | 后续扫描范围 |
|------|-------------------|-------------|
| 并发数 | 1 | 1, 2, 4, 8, 16, 32, 64 |
| 输入长度 (prompt tokens) | ~128 | 64, 128, 256, 512, 1024 |
| 输出长度 (max_tokens) | 256 | 64, 128, 256, 512 |
| 请求速率 (req/s) | 不限速 (burst) | 1, 2, 4, 8, ∞ |

## 本阶段不处理的问题

- 多模型横向对比（不同模型家族性能差异）
- 多卡部署与 tensor parallelism > 1
- LoRA / Adapter 动态加载
- 不同量化方案（AWQ, GPTQ, SmoothQuant）对比
- 自动调参或搜索最优服务配置
- vLLM 源码修改或自定义调度策略
- 与其他推理框架（TGI, TensorRT-LLM）的对比

## 模型选择理由

选择 **Qwen2.5-3B-Instruct-AWQ** 作为 baseline 0 模型：
1. AWQ 4-bit 量化后约 2.69GB，在 RTX 4060 Laptop 8GB 显存上可稳定加载
2. 社区活跃，vLLM 兼容性好，AWQ 量化支持成熟
3. 中文性能优秀，适合后续扩展到中文场景
4. HuggingFace / hf-mirror 可直接下载，无需额外授权

> **注**：原计划使用 7B fp16 模型，但受限于 8GB 显存，实际采用 3B AWQ 量化版本。
